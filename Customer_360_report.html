Objective

The goal of this work was to build a customer360 output table that links each customer to conversion activity, weekly order behaviour, and cumulative revenue over time.

Steps undertaken

  1. Validated the data granularity of orders dataset and made assumptions
    • We first investigated the orders dataset to confirm the intended level of detail in the data.
    • The analysis showed that each order contains only one product and one quantity of that product, so there was no need to aggregate further at the order-product level.
    • We verified that order_id and order_number have a 1:1 relationship, which allowed us to safely ignore order_id in the final query logic.
    • We verified whether we could consider the first order’s week as the same as the first conversion week. Although our check returns a row for customer 9999, we notice that this could be a bad entry. We assume the customer table is well maintained and that this company’s business logic should in fact have first_order_week = first_conversion_week.
  1. Prepared the base customer and conversion structure
    • We used the conversions table as the starting point and joined it to the customer_dimension table to create the foundation for the customer-level view.
    • A right join was used so that customers without conversions would still appear in the output, which enabled us to identify potential customers who had not yet converted.
    • The date_dimension table was used to derive conversion week and next conversion week.
  1. Built the reporting spine
    • We generated a weekly spine for each conversion period so that the final dataset had one row per week between the conversion week and the next conversion week.
    • During development, we encountered issues with the sequence logic for the week counters. To debug this, we introduced a lifetime_week_counter to validate that the spine was being generated correctly.
  1. Joined transactional history
    • We pre-aggregated weekly order data to avoid fan-out during joins and to make the reporting logic more manageable.
    • The weekly spine was then joined to the weekly order aggregates so that each reporting week could show order activity and revenue, even if no purchase occurred in that week.
  1. Populated the output table
    • We inserted the final query results into the output table and used CASE WHEN logic to replace missing values with 0 for weeks where a customer did not place an order.
    • To support repeat refreshes, we used DROP TABLE IF EXISTS then CREATE TABLE before inserting the latest results, recreating the table structure, columns, data types, indexes, and permissions to make sure they match the latest requirements in case these change.

Challenges faced

Challenge 1: Conversion/customer mismatches

A challenge involved unusual records where a conversion appeared to belong to a customer ID that was not present in the customer_dimension table. For example, conversion data for customer 9999 was encountered. We decided not to include these records in the final customer360 output because we assumed the customer table was the authoritative source of valid customer information. Similarly, Maggie (customer_id: 555) had no conversion history, but we chose to include her because she provides useful information about a potential future conversion opportunity.

Challenge 2: Spine generation complexity

Creating the weekly spine was the most technically difficult part of the process. The week_counter logic was initially unstable, and it took time to ensure that the rows were generated in a way that correctly represented the conversion timeline. Adding a lifetime_week_counter helped validate whether the spine logic was working before moving forward.

Challenge 3: Missing purchase weeks

When joining the weekly spine to weekly order aggregates, weeks with no purchases returned NULL values. This would have made the output less interpretable. The issue was resolved by using CASE WHEN expressions such as CASE WHEN aggregate_value IS NULL THEN 0 ELSE aggregate_value END so that missing weeks were represented as 0 revenue and 0 purchases.
